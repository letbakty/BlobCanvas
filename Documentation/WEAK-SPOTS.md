# BlobCanvas — слабые места (аудит 2026-07-17)

Находки код-ревью, ещё НЕ исправлены. Пробелы из
[ARCHITECTURE.md](ARCHITECTURE.md) («Improvements / known gaps» —
Metal-транслюцентность, live `CAMetalLayer`, tiling) здесь не повторяются.

Серьёзность: 🔴 сломает прод · 🟠 сломает при неудачном стечении ·
🟡 качество/перф · ⚪ косметика/долг.

## 🟠 Высокие

### 🟠 B1. Гонка асинхронных сохранений — на диск ложится устаревший блоб
`Persistence/Drawing.swift` (`save(_:)`, `save(_:thumbnailMaxDimension:)`).
Каждый вызов кодирует сессию в `Task.detached` и присваивает
`compressedData` на main actor без сериализации/поколений. **Сценарий**:
автосейв после штриха N (LZFSE медленный на большом рисунке) → штрих N+1 →
второй автосейв кодируется быстрее, пишется первым → затем завершается
первый и **перетирает блоб более старым состоянием**; при выходе до
следующего сейва штрих потерян. **Рецепт**: номер поколения, зафиксированный
до `Task.detached`; отбрасывать присваивание, если поколение не последнее.

### 🟠 B2. `snapshot()` молча возвращает пустую сессию → автосейв обнуляет рисунок
`Views/BlobCanvasView.swift` — `engineView` weak; после ухода view с экрана
`snapshot()` отдаёт пустую `DrawingSession()` (1024×768). **Сценарий**:
экран рисования закрывается, debounced-автосейв срабатывает после dealloc
view → `drawing.save(controller.snapshot())` пишет пустой блоб поверх
реального. Тихая полная потеря данных. **Рецепт**: `snapshot()` →
`DrawingSession?` (nil без view) или кэш последней реальной сессии в
контроллере.

### 🟠 B3. Декомпрессионная бомба в кодеке — OOM до sanity-проверок
`Serialization/DrawingBlobCodec.swift` (`decompressPayload`, `decodeFrame`).
`(compressed as NSData).decompressed(using: .lzfse)` разворачивает весь
payload одним аллоком без лимита; `pointSanityLimit`/`reserveCap`
проверяются только ПОСЛЕ. **Сценарий**: враждебный/битый блоб в единицы МБ
(из iCloud-синка) раздувается LZFSE в гигабайты и убивает процесс jetsam'ом
— при том что README декларирует «hostile decode must never OOM»;
`CodecSafetyTests` этот случай не покрывает. **Рецепт**: стриминговая
`compression_stream`-декомпрессия с потолком выхода (напр. 256 МБ).

## 🟡 Средние

### 🟡 B4. Инкрементальный энкодер: равный граничный штрих «подтверждает» устаревшие sealed-кадры
`Serialization/IncrementalDrawingEncoder.swift` — валидность sealed-кадров
проверяется по одному штриху `sealedCount-1`. **Сценарий**: undo за границу
seal, дорисовка до прежнего количества, где новый граничный штрих
побайтово равен старому (реально для программных штрихов: штамп/фигура/
реплей) → старые кадры переиспользуются, фактические штрихи 0…sealedCount-2
в блобе устаревшие: тихая подмена содержимого. **Рецепт**: хранить и
сравнивать хэш всего sealed-префикса, а не последний штрих.

### 🟡 B5. iOS: поздние точные значения силы Pencil теряются после коммита штриха
`Views/CanvasEngineView.swift` — `touchesEstimatedPropertiesUpdated` патчит
`liveStroke`, но `endStroke` чистит `estimationMap`, а коммит — копия в
`session`. **Сценарий**: обновления `.force` после `touchesEnded` (типично
для последних 1–2 точек) отбрасываются → сохранённый рисунок хранит
оценочное давление, ширина кончика «плывёт» при перерисовке/реплее.
**Рецепт**: патчить последний штрих по сохранённой карте ещё короткое окно
(до `estimatedPropertiesExpectingUpdates.isEmpty`).

### 🟡 B6. Ввод до layout: точки пишутся в координатах view без клампа
`Rendering/CanvasViewport.swift` + `CanvasEngineView.swift` — при
`viewBounds == .zero` → `fitScale == 0` → `viewToCanvas` возвращает точку
как есть (guard `scale > 0` отдаёт `p`). **Сценарий**: событие ввода/
программный тест до первого layout → штрих сохраняется в экранных
координатах, оказывается не там после нормального layout. **Рецепт**: в
`beginStroke` игнорировать ввод, пока `viewport().scale <= 0`.

## ⚪ Долг

### ⚪ B7. `IncrementalDrawingEncoder` не потокобезопасен, а док толкает его в фоновый автосейв
`IncrementalDrawingEncoder.swift` — mutable `cache` без синхронизации, класс
не `Sendable`; док-коммент «call from your debounced auto-save» + паттерн
`Drawing.save` (Task.detached) провоцируют конкурентные `encode`. **Рецепт**:
пометить «только с одного исполнителя» + `assert`, либо сделать actor.

### ⚪ B8. Перестройка буферов посреди штриха стирает live-превью
`CanvasEngineView.swift` — `layout` во время `isStroking` (поворот, resize)
пересоздаёт буферы и зовёт `rebake()`, чистящий `live`: нарисованная часть
текущего штриха визуально исчезает до коммита (точки не теряются).
**Рецепт**: пропускать `ensureBuffers`, пока `isStroking`, перестраивать в
`endStroke`.

### ⚪ B9. `DrawingPlayer`: отрицательный `interStrokeGap` и немонотонные timestamps
`Rendering/DrawingPlayer.swift` — `interStrokeGap` публичный var без клампа
(отрицательный ведёт `cursor` назад); бинарный поиск точек ждёт
монотонности timestamps, которую декодер v2+ не гарантирует на битом блобе →
реплей молча показывает неверный префикс. Только визуальный эффект.
